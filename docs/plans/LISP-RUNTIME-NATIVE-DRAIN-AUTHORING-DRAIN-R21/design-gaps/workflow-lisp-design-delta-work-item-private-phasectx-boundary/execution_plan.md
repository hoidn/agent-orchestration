# Design Delta Work-Item Private Phase Context Boundary Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Do not create a git worktree; this repo's `AGENTS.md` forbids worktrees for this repository.

**Goal:** Restore the blocked-recovery compatibility update to `state/run_state.json` for the Design Delta work-item route while preserving the repaired private `PhaseCtx` boundary and without adding any new authored or hidden bridge carrier.

**Architecture:** Keep `phase-ctx__work-item` as the only private runtime binding for the public `run-work-item` owner boundary. Reuse the already-checked compiled/private `run_state_path` compatibility-bridge lane that reaches the blocked-recovery worker today, and keep runtime-native `record-blocked-recovery-outcome` as semantic authority. The blocked-recovery helper must merge the legacy YAML-era compatibility fields into `state/run_state.json` from the same typed blocked-recovery facts, while `run-selected-item-stdlib` continues to project only typed `SelectedItemResult` behavior and never gains a public `run_state_path`, a new hidden bridge input, or a returned bridge target.

**Tech Stack:** Workflow Lisp `.orc` modules, Design Delta checked manifests under `workflows/examples/inputs/workflow_lisp_migrations/`, runtime-native transition declarations, and focused `pytest` feasibility/build/metadata verification.

Observed failure sequence on July 1, 2026:

- with the tracked `work_item_bridge_support.orc` modules deleted, both
  selectors fail with `FileNotFoundError`;
- after restoring `workflows/library/lisp_frontend_design_delta/work_item_bridge_support.orc`
  and its fixture mirror
  `tests/fixtures/workflow_lisp/valid/design_delta_work_item_runtime/lisp_frontend_design_delta/work_item_bridge_support.orc`
  (the prior blocked attempt performed this restore), both selectors fail at
  compile time with `procedure_arity_mismatch` at the `stdlib_adapters.orc`

  declares 5 positional parameters while the caller passes 4;
- one further latent mismatch sits on the same lane and will surface next:
  `drain.orc` calls `record-drain-terminal-outcome-stdlib` with 2 arguments
  while the callee declares 3.

The cause is a partially landed run-state-carrier retirement: committed
`types.orc` and the narrowed caller modules (`stdlib_payloads.orc`,
`stdlib_adapters.orc`, `drain.orc`) already dropped the parent-drain
`run_state_path` carrier, while `transitions.orc` still declares the leading
`run_state_path` parameter on exactly those two helpers. Restoring the callers
to the old carrier shape cannot re-establish a baseline, because the committed
`DesignDeltaDrainCtx` no longer declares `run_state_path`.

The implementation architecture identifies the real semantic regression that
appears after this baseline damage is repaired:

- blocked-recovery routes return the correct typed terminal result and public
  progress-report path;
- the YAML-era consumer still reads `state/run_state.json`;
- that file no longer receives `blocked_recovery_reason` and
  `blocked_recovery_summary`; and
- the checked smokes then fail with `KeyError: 'blocked_recovery_reason'`.

Execution must therefore do two things in order:

1. restore a compiling baseline for the Design Delta family, per the
   architecture's Workspace Baseline Reconciliation contract (restore the
   deleted bridge-support modules, then reconcile the two carrier-less
   `transitions.orc` helpers with their already-narrowed callers); then
2. repair the missing blocked-recovery compatibility merge without reopening
   the boundary design.

## Causal Failure

The failure is not another `PhaseCtx` bootstrap bug.

The accepted architecture already proves that:

- the public `run-work-item` surface hides `PhaseCtx` behind
  `phase-ctx__work-item`;
- direct entry and imported selected-item routes already converge on the same
  private runtime binding;
- the blocked-recovery worker already computes the right classifier reason and
  public progress report path; and
- the route already carries a hidden compiled/private `run_state_path`
  compatibility lane.

The causal defect is narrower:

- `route-blocked-implementation` still records blocked recovery through the
  low-level helper;
- the low-level helper currently records only the runtime-native
  `drain-run-state` update;
- the helper no longer merges the same typed blocked-recovery facts into the
  YAML-era `state/run_state.json` compatibility document; and
- because that merge was lost, the named live consumer now reads a document
  that lacks `blocked_recovery_reason` and `blocked_recovery_summary`.

The fix must therefore restore the sanctioned compatibility write at the
existing low-level helper path. Do not create a new owner-private return value,
do not add a new hidden carrier, and do not move this behavior into a public
workflow signature.

A separate, purely mechanical defect sits in front of that fix: the
half-finished carrier retirement described under Fresh Failure Evidence. Its
repair direction is fixed by the architecture's Workspace Baseline
Reconciliation section — the narrowed callers are authoritative, the two
carrier-less `transitions.orc` helpers drop their leading `run_state_path`
parameter, and their transitions commit through a state-layout-backed
runtime-native resource instead of the bridge-backed one. The bridge-backed
`drain-run-state` declaration stays: the work-item blocked-recovery lane this
plan restores depends on it.

- Do not reintroduce public `PhaseCtx`, `RunCtx`, generated roots, or checkpoint paths on `run-work-item`.
- Do not add a public `run_state_path` to `run-work-item`, `run-selected-item-stdlib`, `run-work-item-pending`, or any imported callable boundary.
- Do not add a new hidden compatibility-bridge input or a returned bridge-target payload.
- Do not widen the `std/drain::backlog-drain` `run-item` workflow-ref shape.
- Do not reread reports, pointer files, `state/run_state.json`, or native
  drain-run-state backing files to reconstruct blocked-recovery facts after the
  worker returns.
- Do not revive `update_lisp_frontend_run_state.py`,
  `record_lisp_frontend_blocked_recovery_outcome.py`, inline Python/shell glue,
  or any uncertified script fallback.
- Do not change shared runtime files such as
  `orchestrator/workflow/executor.py`,
  `orchestrator/workflow/transition_executor.py`,
  `orchestrator/workflow_lisp/phase.py`, or
  `orchestrator/workflow_lisp/lowering/workflow_calls.py` unless the
  implementation first proves the accepted `.orc` and checked-metadata shape is
  insufficient. If that happens, stop and reopen design instead of silently
  widening the slice.
- Keep the repair generic to the sanctioned compatibility-bridge mechanism. Do
  not add a Design Delta step-name special case to shared runtime or lowering
  code.
- Treat the narrowed carrier-less call shapes in `stdlib_adapters.orc`,
  `stdlib_payloads.orc`, and `drain.orc` as authoritative. Do not re-add
  `run_state_path` carrier arguments there, and do not restore the retired
  `DesignDeltaDrainCtx.run_state_path` field, to resolve the arity mismatch.
- Do not make ordinary drain or gap-drafting routes write
  `state/run_state.json` while rehoming the two parent-drain transitions, and
  do not hand-author the legacy path inside `transitions.orc`.
