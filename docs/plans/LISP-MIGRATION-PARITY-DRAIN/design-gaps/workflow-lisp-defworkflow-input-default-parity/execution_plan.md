# Workflow Lisp Defworkflow Input-Default Parity Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Do not create a git worktree; this repo's `AGENTS.md` forbids worktrees. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add authored `:default` support on `defworkflow` boundary parameters, normalize supported literal defaults into the existing workflow input contract `default` field, preserve caller-overrides-default precedence for entry workflows and imported workflow calls, and keep the current autogenerated top-level `PhaseCtx` defaults only as a compatibility fallback.

**Architecture:** Keep this slice inside `orchestrator/workflow_lisp/`. Parsing, boundary validation, imported-bundle signature reconstruction, call typecheck, and call lowering all need to understand one new concept: workflow-boundary default metadata that stays separate from the existing `signature.params` tuple shape. Store normalized authored defaults on `WorkflowSignature`, attach them onto flattened public workflow input contracts in `orchestrator/workflow_lisp/contracts.py`, reconstruct the same metadata from imported bundles, and let workflow-call typecheck/lowering omit only those callee bindings whose compiled contract already has a default. Runtime binding logic under `orchestrator/workflow/` remains unchanged and is proven by tests rather than modified.

**Tech Stack:** Python 3, Workflow Lisp frontend compiler/lowering, typed loaded bundles, shared workflow runtime input binding, `pytest`

## Hard Scope Limits

Implement only this bounded slice:

- add authored `:default` syntax on `defworkflow` parameters;
- validate and normalize supported default literals during workflow-boundary type resolution;
- store authored defaults as frontend boundary metadata on compiled workflow signatures;
- lower authored defaults onto the existing workflow input contract `default` field for public inputs only;
- reconstruct the same default metadata for imported workflow signatures from compiled public input contracts;
- allow workflow calls to omit callee bindings only when the callee signature metadata proves that a compiled default exists for that parameter;
- keep entry-workflow binding and imported workflow call binding semantics unchanged by reusing the existing runtime precedence;
- subordinate the current autogenerated top-level `PhaseCtx` defaults so they apply only when no authored default exists;
- extend focused parser/typecheck/lowering/build/runtime tests and fixtures for authored defaults.

Explicit non-goals:

- no defaults on `defproc`, `defun`, local bindings, record fields, workflow `call` `with:` bindings, or collection elements;
- no new call-site syntax for "use default"; omission of a binding is the only supported path, and only for already-defaulted callee params;
- no dynamic, effectful, environment-derived, record-valued, collection-valued, `nil`, `WorkflowRef`, or `ProcRef` defaults;
- no changes to `orchestrator/workflow/signatures.py`, `orchestrator/workflow/calls.py`, `specs/dsl.md`, `specs/io.md`, runtime command execution, source-map schemas, Core AST, Semantic IR, or Executable IR unless a new focused failing test proves an existing contract bug outside the authored-default path;
- no review-loop, findings, command-result write-root, `resume-or-start`, reusable-state, parity-report, or YAML-deprecation work.

## Required Contract Deltas

These are fixed implementation decisions for this slice:

- `defworkflow` accepts exactly two parameter shapes:
  - `(name Type)`
  - `(name Type :default <literal>)`
- No other workflow-parameter keywords are introduced.
- Supported authored defaults in this slice are only for parameters whose resolved boundary type lowers to exactly one flattened workflow input contract.
- Supported normalized default kinds are:
  - `String` and relpath/path boundary types from authored string literals
  - `Int` from integer literals
  - `Float` from float literals
  - `Bool` from boolean literals
  - enum boundary types from authored enum-member symbols normalized to the member string stored in workflow contracts
- Unsupported defaults that must fail in the frontend include:
  - record, union, collection, `nil`, `WorkflowRef`, `ProcRef`, or effectful/default-expression shapes
  - defaults on parameters that flatten to multiple workflow inputs, including `PhaseCtx`, `ItemCtx`, `DrainCtx`, and similar structured boundaries
  - malformed parameter keyword syntax, unknown keywords, or `:default` without a value
- `WorkflowSignature.params` remains unchanged, but `WorkflowSignature` gains a separate default-metadata channel that survives local workflow definitions, imported-bundle reconstruction, procedure-generated private workflow wrappers, and workflow specializations.
- Imported workflow signatures reconstruct default metadata from the compiled public input contracts' existing `default` fields; no second serialized default channel is introduced.
- Authored defaults attach only to public authored workflow inputs.
- Authored defaults never attach to compiler-generated internal inputs such as managed write roots.
- If both an authored default and the old top-level `PhaseCtx` compatibility helper could populate the same flattened input, the authored default wins and the compatibility helper is skipped for that field.
- The flattened input `contract_definition["default"]` remains the sole serialized authority for workflow-boundary defaults.
- A workflow call may omit a callee binding only when the callee `WorkflowSignature` metadata says the parameter has a compiled default; missing non-defaulted bindings still fail in the frontend.
- Lowering omits missing defaulted bindings from the generated `with:` map so the existing runtime call binder, not the frontend, applies the callee contract default.

## Notes For The Implementer

- Prefer inline temp modules in tests where they keep the regression local and easy to read; create fixture files only when the same authored-default shape is reused across lowering/build/runtime tests.
- Keep authored-default validation close to the existing workflow-boundary checks so unsupported defaults fail before body typechecking or lowering.
- If you need a helper to determine whether a type lowers to exactly one input field, reuse `derive_workflow_boundary_fields(...)` or a tightly scoped wrapper instead of inventing a second flattening algorithm.
- Do not duplicate path-safety or existence checks in the frontend. The frontend only proves literal/type compatibility and single-field eligibility; shared validation still owns relpath policy.
- Do not add a frontend-only "apply default" branch for workflow calls. The only frontend responsibility is to preserve default metadata and omit already-defaulted callee bindings from lowered `with:` maps so the existing runtime contract binder stays authoritative.
