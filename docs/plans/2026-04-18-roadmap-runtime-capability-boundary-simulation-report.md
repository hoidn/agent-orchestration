# Roadmap Runtime Capability Boundary Simulation Report

## Decision

Recommendation: `ADOPT_NARROWLY`

Apply the proposed runtime-capability-boundary language to the major-project roadmap draft and review prompts only. The current prompt set is a clear improvement over the original EasySpin prompts because it asks for a completion boundary, omitted in-scope work, end-to-end acceptance gates, and stronger review against subset/pilot narrowing. That probably reduces the chance of the CUDA miss, but it does not reliably force the roadmap to distinguish "GPU readiness/profiling evidence" from "a public device execution path users can invoke." The candidate lines are targeted enough to close that gap for technology-port roadmaps without making CUDA mandatory or adding noise to non-port projects.

## Analysis

The candidate would likely move the CUDA issue earlier into roadmap drafting or roadmap review. In Scenario A, the original prompts demonstrably allowed a roadmap to mention GPU only as "where available" benchmarking and readiness notes. The current prompts make that less likely by asking for completion boundaries and omitted in-scope work, but they still do not name runtime capability dimensions. The improved prompt makes public execution path, device, dtype, precision, batching, and performance claims explicit things to assign.

The current prompt set does not fully solve the issue. It gives a skeptical reviewer enough conceptual footing to catch the miss, but it leaves too much discretion. A reviewer focused on sequencing, manifest validity, and subset boundaries could still approve a roadmap that treats CUDA as an optimization/reporting topic rather than a user-visible runtime capability.

The candidate does not turn CUDA into a universal requirement. It says material runtime dimensions implied by the brief must be assigned somewhere, and the allowed destinations include explicit blocker and explicit non-goal. In Scenario B, that should surface backend/public-execution semantics without hallucinating GPU. In Scenario C, the condition should not fire because a read-only dashboard is not a technology-port roadmap.

The main regression risk is checklist gravity: agents may add a runtime-capability paragraph even for small backend migrations. The risk is acceptable because the line is conditional, short, and outcome-focused. It does not ask for exhaustive matrices, report ledgers, or implementation details.

No downstream design, plan, or implementation prompt edit is required for this specific improvement. The failure was at the roadmap boundary: the project map failed to assign an implied capability dimension. Once the roadmap assigns that dimension, existing tranche design/plan prompts can handle the selected tranche.

## Recommended Edit

Add this to `workflows/library/prompts/major_project_stack/draft_project_roadmap.md` after the completion-boundary paragraph and before `The roadmap must:`:

```text
For technology-port roadmaps, identify material runtime capability dimensions implied by the brief, such as backend, device, dtype, precision, batching, public execution path, and performance claims. Each material dimension must be assigned to an implementation tranche, an acceptance gate, an explicit blocker, or an explicit non-goal.
```

Add this to `workflows/library/prompts/major_project_stack/review_project_roadmap.md` near the completion-boundary and omitted-in-scope-work checks:

```text
- a technology-port roadmap leaves a material runtime capability dimension implied by the brief, such as backend, device, dtype, precision, batching, public execution path, or performance claim, unassigned to a tranche, acceptance gate, explicit blocker, or explicit non-goal
```

Do not add matching language to tranche design, plan, implementation, or implementation-review prompts yet. Re-evaluate only if a future tranche design treats an assigned runtime capability as vague performance documentation instead of an implementation or acceptance boundary.
