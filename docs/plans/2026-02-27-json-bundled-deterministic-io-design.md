# ADR: Structured Deterministic I/O Bundles for Workflow DSL

**Status:** Proposed  
**Date:** 2026-02-27  
**Owners:** Orchestrator maintainers

## Context

Current deterministic dataflow is reliable but fragmented:

1. Step outputs are contract-validated from many one-value files (`expected_outputs[*].path`).
2. Runtime stores typed output values in `steps.<Step>.artifacts`.
3. `publishes/consumes` use that typed map and ledger metadata.
4. `relpath` consumes also materialize pointer files.

This model is correct but operationally noisy. Workflows that exchange several deterministic values often create many tiny `state/*.txt` files, even when the producer naturally emits structured JSON.

## Problem Statement

We need to reduce deterministic I/O file sprawl without weakening:

1. Type validation and enum constraints.
2. Path safety (`under`, `must_exist_target`).
3. Publish/consume freshness guarantees.
4. Backward compatibility for existing `version: "1.2"` workflows.

## Decision

Add an opt-in `version: "1.3"` I/O contract for JSON bundles while preserving existing v1.2 behavior.

### D1. Add `output_bundle` for deterministic step outputs

```yaml
steps:
  - name: RunTargetedChecks
    command: ["bash", "-lc", "python scripts/run_checks.py"]
    output_bundle:
      path: artifacts/checks/latest-summary.json
      fields:
        - name: check_log_path
          json_pointer: /check_log_path
          type: relpath
          under: artifacts/checks
          must_exist_target: true
        - name: failed_count
          json_pointer: /failed_count
          type: integer
```

Semantics:

1. `output_bundle.path` points to a single JSON document generated by the step.
2. Each `fields[*]` declares one deterministic artifact (same contract surface as `expected_outputs` entries).
3. Runtime extracts, validates, and writes typed values into `steps.<Step>.artifacts`.
4. `publishes.from` still references artifact names (`check_log_path`, `failed_count`) exactly as today.

### D2. Add `consume_bundle` for deterministic step inputs

```yaml
steps:
  - name: ReviewImplVsPlan
    provider: codex
    consumes:
      - artifact: plan
        policy: latest_successful
      - artifact: execution_log
        policy: latest_successful
    consume_bundle:
      path: state/consumes/review_impl_vs_plan.json
      include: ["plan", "execution_log"]  # default: all consumed artifacts
```

Semantics:

1. After consume resolution succeeds, runtime can write one JSON object containing selected consumed values.
2. `consume_bundle` is convenience I/O; provenance/freshness rules remain entirely in `consumes`.
3. Existing pointer materialization remains unchanged for `relpath` artifacts (backward compatibility).

### D3. Keep legacy contracts fully supported

1. `expected_outputs` remains valid.
2. Pointer-file flows remain valid.
3. `output_bundle` and `consume_bundle` are additive and optional.

### D4. Use asymmetric strictness by step role

To avoid over-constraining heavy LLM work while keeping control flow deterministic:

1. Heavy implementation/fix steps (`ExecutePlan`, `FixIssues`) should stay flexible:
   - `output_capture: text` (or `lines`),
   - minimal deterministic contracts (for example one session-log artifact path),
   - no rigid JSON-shape requirements for full reasoning output.
2. Assessment/review/gate steps (`AssessExecutionCompletion`, `ReviewImplVsPlan`, gate checks) should be strict:
   - `output_capture: json`,
   - `allow_parse_error: false`,
   - deterministic artifact contracts for decision fields.
3. Branching/control-plane decisions must consume published review/assessment artifacts only, never raw prose logs.

This keeps model-heavy execution unconstrained while enforcing deterministic machine contracts where they matter.

## Why This Is Not a Pointer-Model Collision

Pointers remain canonical for `relpath` artifacts. Bundles are derived transport formats:

1. Output bundle: one-to-many extraction into the same typed artifact map already used by publish/consume.
2. Consume bundle: many-to-one materialization of already-resolved consumed values.

Canonical dataflow truth stays in typed artifacts plus artifact version ledger, not in prompt text.

## Validation and Runtime Rules

Loader (v1.3+):

1. `output_bundle` and `expected_outputs` are mutually exclusive per step (MVP).
2. `output_bundle.fields` must be non-empty; field names unique.
3. `json_pointer` must be valid RFC 6901 pointer syntax.
4. `consume_bundle.include`, if present, must be a subset of step `consumes` artifact names.

Executor:

1. Step success with missing/invalid output bundle is `contract_violation` (exit `2`).
2. Field-level validation reuses current output-contract validator logic.
3. `consume_bundle.path` is written only after consume contract passes.
4. Publish/consume ledger behavior is unchanged.

## State Model

No `state.json` schema migration required:

1. Continue writing typed artifacts to `steps.<Step>.artifacts`.
2. Continue recording lineage in `artifact_versions` and `artifact_consumes`.
3. Optional debug metadata can note bundle extraction/materialization paths.

## Migration Plan

Phase 1 (ship feature, no workflow breakage):

1. Implement loader/executor support for `output_bundle` and `consume_bundle`.
2. Add tests for bundle extraction, validation, and publish/consume interoperability.

Phase 2 (adopt in high-noise workflows):

1. Migrate command-heavy steps from many `state/*.txt` outputs to one JSON output bundle.
2. Migrate assessment/review/gate steps first to strict JSON contracts.
3. Keep heavy execution/fix steps on flexible text outputs unless a concrete control-plane need requires more structure.
4. Migrate selected consumer steps to one consume bundle where it improves clarity.

Phase 3 (documentation):

1. Update `specs/dsl.md` with v1.3 fields and compatibility guidance.
2. Add an example workflow showing before/after fragmentation reduction.

## Alternatives Considered

1. Keep file-per-value only: rejected (high operational noise).
2. Replace pointer model entirely: rejected (breaks compatibility, higher migration cost).
3. Single global run JSON for everything: rejected (couples unrelated steps, weak isolation).

## Test Plan

1. Loader tests: v1.3 gating, field validation, subset validation for `consume_bundle.include`.
2. Runtime tests: successful extraction, type/enum/path failures, publish/consume compatibility.
3. Integration tests: migrated example workflow with fewer pointer files and unchanged decision behavior.
4. Backward compatibility tests: existing v1.2 workflows unchanged.
5. Policy tests: control flow decisions derive from strict assessment/review artifacts, not free-form execution logs.
