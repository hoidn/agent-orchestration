# Dashboard Observability Summary GUI

Status: draft implemented plan

## Goal

Add a read-only dashboard view for run summary observability artifacts so users
can browse provider and phase summaries without knowing call-frame paths.

## Design

The GUI extends the existing stdlib dashboard instead of adding a new frontend
server. The dashboard already provides read-only run pages, safe run-scoped file
preview routes, CSP headers, and route-scoped path validation. Summary browsing
should reuse those surfaces.

Route:

```text
/runs/<workspace_id>/<run_dir>/summaries
```

Data source:

```text
RUN_ROOT/summaries/index.json
RUN_ROOT/summaries/run-summary.md
```

The route reads the aggregate summary index generated by the summary observer.
It does not scan arbitrary summary files as semantic authority. If the aggregate
index is missing, the page reports that no summary hub is available and points
users back to the run detail page.

## Page Shape

The page renders:

- a run header and back link;
- a live current-step panel that updates while the page is open;
- a compact status strip with total summaries and counts by kind;
- a table of summary entries with step name, kind, profile, status, duration,
  frame root, and links to summary, snapshot, and error files;
- an escaped preview of `run-summary.md` when present.

All file links use the existing dashboard file route:

```text
/runs/<workspace_id>/<run_dir>/files/run/<run-relative-path>
```

The page must not expose absolute filesystem paths. Markdown summary content is
rendered as escaped text in a `<pre>` block, not interpreted as HTML.

## Live Current-Step View

The summary hub also exposes a small JSON polling endpoint:

```text
/runs/<workspace_id>/<run_dir>/summaries/live.json
```

The endpoint returns a run-scoped read model containing:

- run status and display status;
- current step name, step id, started time, heartbeat, and age fields;
- latest summary counts;
- the newest summary entry for the current step when one exists.

The page uses a small nonce-protected inline script to poll this endpoint every
few seconds and update only the current-step panel. This keeps the page dynamic
without turning the dashboard into a separate frontend application. The
JavaScript renders text with `textContent` and builds only run-local links
returned by the endpoint.

If JavaScript is disabled, the static summary hub remains useful and the page
can still be refreshed manually.

## Live Agent-Output Notes

The current-step panel should not treat raw streamed agent output as workflow
state. Raw provider transport is high-volume, provider-specific, and may contain
partial or redundant text. When live narration over agent output is needed, a
runtime-side observer or sidecar should:

- read a bounded tail of the current provider transport spool;
- call a configured cheap summary provider at a throttled interval;
- write the result as a run-local observability artifact;
- leave `state.json` current-step metadata as the authority for what is running.

The dashboard may display or link that generated live note once the artifact
exists, but it should not call providers directly from page requests or polling
handlers.

Initial runtime surface:

```text
--live-agent-notes
--live-agent-note-provider <provider>
--live-agent-note-interval-sec <seconds>
--live-agent-note-timeout-sec <seconds>
--live-agent-note-max-tail-chars <chars>
```

These flags configure an optional observer under the existing
`step_summaries` runtime observability block. The observer runs only while a
session-enabled provider step has a retained transport spool. It writes:

```text
RUN_ROOT/summaries/live-current-step.md
RUN_ROOT/summaries/live-current-step.json
```

The Markdown file is a short human-facing note. The JSON file is display
metadata: source step, visit count, generation time, and any best-effort error
state. Both files are observability artifacts only. They are not workflow state
and do not affect routing.

## Run Detail Integration

The run detail page shows a prominent "Summary Hub" link when
`RUN_ROOT/summaries/index.json` exists. The existing Observability Files section
also lists summary hub files as ordinary run-scoped observability files.

## Error Handling

- Missing `summaries/index.json`: render a 200 page with a clear empty state.
- Invalid JSON or wrong schema shape: render a 200 page with a warning and raw
  file preview link.
- Missing target summary/snapshot/error files: render their links only when the
  corresponding path is present in the index; the existing file route displays
  missing-file state.
- Unsafe paths in index entries: do not link them; show them as unavailable.

## Security

Dashboard summary routes remain read-only. They must not execute workflow
commands, resume runs, call providers, mutate state, or parse summary markdown
as HTML. Route paths are validated through `FileReferenceResolver.run_ref`.

The live endpoint returns JSON only and includes no absolute paths. The summary
hub page may loosen `script-src` only with a per-response nonce for its own
inline updater; other dashboard routes keep the default no-script CSP.

## Tests

Add dashboard server tests for:

- run detail links the summary hub when the aggregate index exists;
- `/summaries` renders grouped entries and safe run-file links;
- markdown preview is escaped;
- invalid or missing index files produce safe, non-crashing pages;
- traversal paths in index entries are not linked.
- `/summaries/live.json` returns current-step data and safe latest-summary
  links without absolute paths;
- the summary hub page includes the live panel and nonce-protected updater.
