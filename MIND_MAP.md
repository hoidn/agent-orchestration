[1] **Project Overview** - This repository implements a deterministic YAML workflow orchestrator for multi-agent development pipelines, balancing strict contracts with practical filesystem-driven coordination [2][3]. The core runtime lives in `orchestrator/` (32 Python files, 5,876 LOC), while behavioral confidence is driven by a heavy test suite in `tests/` (37 test modules, 12,189 LOC) and a normative spec set in `specs/` (18 Markdown files, 1,330 LOC) [18][20]. The orchestration model is sequential (no concurrency blocks), with explicit control flow (`when`, `on.*.goto`, `_end`) and queue conventions (`inbox/`, `processed/`, `failed/`) [11][15]. What makes the project distinctive is its acceptance-ID traceability: many code paths and tests are mapped directly to AT requirements in `specs/acceptance/index.md`, so behavior is documented as contract-first rather than implementation-first [21]. Development happened in a tight burst during September 2025, with one large code import followed by cleanup and packaging stabilization [24].

[2] **Spec-First Contract Model** - The authoritative interface is defined by the sharded spec suite in `specs/index.md` and companion normative docs like `specs/dsl.md`, `specs/providers.md`, and `specs/state.md`, while implementation details are expected to conform to these contracts [1][6]. The DSL track (`version: "1.1"` and `"1.1.1"`) is intentionally independent from state schema track (`schema_version: "1.1.1"`), allowing runtime state evolution without forcing DSL churn [13][25]. Strict validation is a design principle: unknown fields are rejected at load time, feature gating is version-enforced (`depends_on.inject` requires 1.1.1), and invalid path patterns fail fast with exit code 2 [6][23]. This creates a high-signal boundary between authoring errors and execution errors, which improves resume semantics and troubleshooting [14][17]. The acceptance list in `specs/acceptance/index.md` acts as a living requirements index and ties directly into test module naming conventions [18][21].

[3] **Execution Lifecycle Data Flow** - A typical run starts with CLI parsing (`orchestrator/cli/main.py`) and workflow validation (`orchestrator/loader.py`), then initializes run state and logs under `.orchestrate/runs/<run_id>/` via `StateManager` in `orchestrator/state.py` [4][6][13]. `WorkflowExecutor.execute()` in `orchestrator/workflow/executor.py` drives ordered step execution, evaluating conditional guards, branching, retries, and loop semantics as it mutates `state.json` [5][11][12]. Command steps route through `StepExecutor.execute_command()` and provider steps route through `ProviderExecutor`, but both converge into shared output-capture and persistence behavior [8][9]. Dependency resolution/injection can mutate the in-memory prompt (not source files) before provider invocation, and wait-for steps block on file globs while recording poll metrics [10][15]. Debug mode layers prompt audits and state backups over this baseline flow, giving deterministic replay artifacts for incident review [17].

[4] **CLI Surface and Operator Workflow** - User-facing entry points are `orchestrate run` and `orchestrate resume`, defined in `orchestrator/cli/main.py` and implemented in `orchestrator/cli/commands/run.py` and `orchestrator/cli/commands/resume.py` [1][3]. `run` handles context composition (`--context`, `--context-file`), safety rails (`--clean-processed`, `--archive-processed`), dry-run validation, retry policy defaults, and debug/logging toggles [15][22][23]. `resume` reloads existing run state, validates workflow checksum, optionally repairs from backups, and supports force-restart semantics for starting fresh despite state drift [13][14]. CLI safety enforcement matters operationally: cleanup and archive destination checks prevent accidental destructive actions outside workspace boundaries [23]. The CLI surface is intentionally minimal but wired to rich runtime behavior and log/state artifacts that operators can inspect immediately [17].

[5] **Runtime Engine Composition** - `WorkflowExecutor` in `orchestrator/workflow/executor.py` is the central coordinator and composes specialized subsystems rather than embedding all concerns inline [1][3]. It initializes `ProviderRegistry`/`ProviderExecutor` for LLM tooling, `StepExecutor` for raw commands and wait steps, `DependencyResolver`/`DependencyInjector` for file-context orchestration, `ConditionEvaluator` for `when` predicates, and `VariableSubstitutor` for namespace interpolation [8][9][10][11]. This composition keeps step-type execution paths distinct while preserving unified state updates and control-flow decisions [12][13]. The file is the dominant hotspot at 1,256 LOC, reflecting where cross-cutting behavior (retries, goto, loop scoping, debug audit hooks) converges [20][22]. Most deep changes to execution semantics should start with this node and then branch into subsystem nodes to avoid hidden regressions [6][18].

[6] **Workflow Loading and Validation** - `WorkflowLoader` in `orchestrator/loader.py` enforces strict DSL schema, supported versions (`1.1`, `1.1.1`), execution-field exclusivity, deprecated field rejection, and goto target integrity before runtime execution begins [2][5]. A custom YAML loader (`PreservingLoader`) disables implicit coercion of keys like `on` to booleans, preserving authored semantics for branching blocks [11]. Validation includes runtime-adjacent safeguards like path checks on declared file fields and proactive rejection of `${env.*}` namespace use in DSL-authored substitutable fields [7][23]. Errors are accumulated as typed `ValidationError` records and thrown as `WorkflowValidationError` with exit code 2, clearly separating authoring failures from execution-time failures [21][22]. Because this module is the contract gate, any DSL extension or version bump should update both validation branches and acceptance coverage in lockstep [25].

[7] **Variable Model and Substitution Engine** - `VariableSubstitutor` in `orchestrator/variables/substitution.py` implements `${...}` resolution across namespaces (`run`, `context`, `steps`, `loop`, `item`), with undefined-variable tracking used for deterministic command/provider failure paths [3][6]. Runtime variable assembly differs by scope: top-level steps use full run state while looped steps can be scoped to current iteration state to enforce AT-65 semantics [12][21]. Substitution is recursive over strings/lists/dicts and supports escaping (`$$`, `$${`) while preserving non-string data in nested structures for provider params [8]. A key design boundary is literal file semantics: input file contents are read as-is and are not templated, so dynamic content must be created in earlier steps or via injection [10][19]. If a behavior appears “mysteriously unsubstituted” or “failing as undefined,” this node is the first diagnostic anchor before control-flow or provider debugging [11][22].

[8] **Provider Templates and Invocation Pipeline** - Provider contracts are defined by `ProviderTemplate`, `ProviderParams`, and `ProviderInvocation` in `orchestrator/providers/types.py`, with lookup/default merging in `orchestrator/providers/registry.py` [5][7]. Built-ins include `claude`, `gemini`, and a stdin-based `codex` template; workflow-defined providers can override or augment via `providers:` config [19]. `ProviderExecutor.prepare_invocation()` merges defaults with step params, performs nested substitution, validates placeholder completeness, resolves secrets/env precedence, and forbids `${PROMPT}` misuse in stdin mode [16][23]. `execute()` then runs subprocess calls with timeout handling and returns byte streams that route into shared output capture/state persistence paths [9][13]. Provider behavior is therefore a layered pipeline: template validity, context substitution, prompt composition/injection, execution, and capture, each with dedicated failure context for observability [10][17].

[9] **Command Execution and Output Capture** - Raw command execution is handled by `StepExecutor.execute_command()` in `orchestrator/exec/step_executor.py`, which executes argv-safe subprocesses (`shell=False`), applies timeout mapping, and integrates secrets-aware environment setup [5][16]. Output semantics are centralized in `OutputCapture` (`orchestrator/exec/output_capture.py`) with mode-specific limits: `text` (8 KiB), `lines` (10,000), and `json` (1 MiB parse buffer), including overflow/error behaviors [3][21]. Tee semantics intentionally preserve full stdout to `output_file` even when state is truncated, while stderr is independently logged under run logs [17]. The module also supports `allow_parse_error` for JSON mode as a controlled degradation path that retains raw output/debug context instead of hard failing [22]. Command and provider paths both consume these capture semantics, which keeps downstream state analysis consistent regardless of step type [8][13].

[10] **Dependencies and Injection System** - Dependency matching is implemented in `orchestrator/deps/resolver.py`, where required/optional glob sets are resolved deterministically with path safety checks and lexicographic ordering guarantees [3][23]. Injection composition lives in `orchestrator/deps/injector.py` and supports `list`, `content`, and `none` modes with prepend/append positioning and ~256 KiB total injected payload cap [2][25]. Content mode annotates each injected file with `=== File: <path> (<shown>/<total> bytes) ===` headers, and truncation metadata can be persisted into step debug state for forensic review [17]. The important boundary is mutability: injection only changes composed prompt text in memory; source files remain unchanged by this subsystem [8]. This design collapses duplicate file-list maintenance and enables DRY prompt templates, which is the primary value proposition of DSL v1.1.1 over v1.1 [19][24].

[11] **Control Flow and Conditional Semantics** - Branching behavior is split between loader validation (shape/targets) and runtime evaluation in `WorkflowExecutor._handle_control_flow()` plus `ConditionEvaluator` (`orchestrator/workflow/conditions.py`) [5][6]. `when.equals`, `when.exists`, and `when.not_exists` determine skip vs execute, with skipped steps recorded as `status=skipped` and `exit_code=0`, preserving deterministic state for resume and diagnostics [12][13]. After step execution, handler precedence is explicit: `on.success`/`on.failure`, then `on.always` override, then strict-flow/`--on-error` policy if no goto applies [22]. `_end` is a reserved termination target and is resolved like other goto pathways after static validation of reachable names [3][21]. If workflow behavior appears non-linear, inspecting this node with execution logs usually resolves whether branching or retries caused the observed path [17].

[12] **For-Each Loops and Pointer Resolution** - Loop execution is implemented in `WorkflowExecutor._execute_for_each()` and pointer parsing in `orchestrator/workflow/pointers.py`, supporting `items_from: steps.<Name>.lines|json.<path>` and literal `items` arrays [3][11]. Per-iteration contexts expose `${item}` and `${loop.*}`, while loop-scoped `steps.*` references are intentionally narrowed to current iteration state to avoid cross-iteration bleed (AT-65) [7][21]. Results are recorded in dual forms: array summary at `steps.<LoopName>` and flattened iteration keys (`steps.<LoopName>[i].<StepName>`) for compatibility and direct lookups [13]. Conditional execution and retries inside loops follow the same primitives as top-level steps, but backup/debug naming includes loop indices to preserve traceability [17][22]. Any bug involving nested step substitution or iteration resume behavior should be debugged through this node together with state snapshots [14].

[13] **State Model and Persistence Guarantees** - `StateManager` in `orchestrator/state.py` owns run identity, checksum tracking, state serialization, atomic writes (`temp + rename`), and backup rotation for debug/backup-enabled runs [3][4]. The canonical record is `.orchestrate/runs/<run_id>/state.json` with `schema_version`, workflow checksum, step map, and loop bookkeeping, matching the normative shape in `specs/state.md` [2]. `StepResult` and `ForEachState` dataclasses normalize stored fields across command/provider/wait/loop pathways, while preserving mode-specific payloads (`output`, `lines`, `json`, wait metrics) [9][12]. Because every execution decision and recovery path consults persisted state, any corruption or drift materially affects resume semantics and must be treated as a first-class runtime concern [14]. This node is also where run-root metadata for `${run.root}` originates, linking variable substitution to durable on-disk run artifacts [7][17].

[14] **Resume, Repair, and Force-Restart Flow** - Resume behavior is implemented in `orchestrator/cli/commands/resume.py`, where state load, optional backup repair, checksum validation, and run-continuation logic are orchestrated before handing back to `WorkflowExecutor` [4][13]. Without `--force-restart`, workflow checksum mismatches block resume to prevent silently continuing with modified definitions; with `--force-restart`, a new run lineage is created and prior state is ignored [22][24]. Repair mode attempts to recover from the newest valid `state.json.step_<Step>.bak`, reflecting the atomic+backup strategy in state management [13]. The runtime also short-circuits already-completed/skipped steps (including loop iterations) when resuming, reducing duplicate side effects in long-running workflows [12]. Operationally, this node is the boundary between safe recovery and unintended replay, so debugging interrupted runs should always begin here and in run logs [17].

[15] **Queue Conventions and Wait-For Primitive** - Queue lifecycle helpers in `orchestrator/fsq/queue.py` implement atomic task creation (`*.tmp` then rename to `*.task`) and explicit move helpers for processed/failed directories, but orchestrator policy keeps lifecycle user-authored rather than automatic [1][3]. Blocking synchronization is provided by `WaitFor` in `orchestrator/fsq/wait.py`, which polls workspace-relative globs with timeout/poll/min-count controls and records files plus timing metrics [13]. Runtime wait checks enforce path safety (reject absolute/parent traversal) and filter symlinks that resolve outside workspace, protecting control loops from escaping filesystem boundaries [23]. Workflow examples under `workflows/examples/wait_for_example.yaml` and queue patterns in `specs/examples/multi-agent-inbox.md` demonstrate intended use in agent handoffs [19]. This node is critical for multi-agent orchestration reliability because it links filesystem conventions, synchronization semantics, and recoverable state evidence [12].

[16] **Secrets and Environment Security Semantics** - `SecretsManager` in `orchestrator/security/secrets.py` enforces declared-secret presence from orchestrator process environment, overlays step `env` values with precedence, and tracks secret material for masking across outputs/logs [5][8]. Missing declared secrets fail steps with exit code 2 and structured `missing_secrets` context, enabling deterministic security failures rather than ambiguous command/provider errors [21]. Masking is best-effort and applied to text/JSON structures and logging filters, reducing accidental credential leakage in debug artifacts while preserving actionable diagnostics [9][17]. Path safety itself is enforced across loader, dependency, and wait-for logic, but child processes remain OS-capability-bound, so isolation beyond DSL guardrails depends on host sandboxing [23]. This node ties policy-level security claims to concrete implementation points and failure signatures used by tests [18].

[17] **Observability and Debug Artifacts** - Run observability converges in `RUN_ROOT/logs/` with `orchestrator.log`, per-step stdout/stderr spill files, prompt audits (`<Step>.prompt.txt` in debug mode), and optional debug payloads in state [3][13]. `WorkflowExecutor._write_prompt_audit()` writes masked composed prompts after dependency injection, which is essential for diagnosing provider behavior without mutating source prompts [8][10][16]. Output capture also contributes observability by preserving full streams when state storage truncates, ensuring postmortems are not constrained by state size limits [9]. Tests such as `tests/test_at70_prompt_audit.py` and `tests/test_at74_e2e_observability.py` extend this layer from unit behavior into operator-facing evidence quality [18][21]. Effective debugging in this project generally means correlating this node’s artifacts with control-flow and substitution nodes instead of relying on CLI return codes alone [11][14].

[18] **Testing Strategy and Confidence Model** - The test suite is large and intentionally acceptance-oriented: most files map to AT IDs (for example `test_output_capture.py`, `test_dependency_resolution.py`, `test_at71_retries_goto.py`) and target behavioral contracts rather than implementation details [1][21]. `tests/README.md` separates unit/integration from opt-in E2E markers (`e2e`, `requires_secrets`) so default runs stay fast and network-free while still supporting real provider validation paths [17]. High-density hotspots include control flow, loops, retries, injection, and safety boundaries, mirroring the complexity concentration in `WorkflowExecutor` and related modules [5][22]. This coverage pattern is useful for contributors: if you touch execution semantics and do not update matching AT-focused tests, regressions are likely even when local behavior seems plausible [24][25]. The suite acts as both guardrail and documentation index when read alongside specs and mind-map links [2].

[19] **Workflow Examples as Operational Playbooks** - Example workflows in `workflows/examples/` demonstrate concrete usage patterns for conditional branching, retries, wait-for synchronization, output capture modes, prompt audit, env literal semantics, and dependency injection [4][15]. These examples are informative rather than normative, but they encode idioms that reduce authoring mistakes, especially for queue-based multi-agent pipelines and status-gated flows [10][11]. The richer design patterns in `specs/examples/patterns.md` and `specs/examples/multi-agent-inbox.md` show how to combine static prompts with dynamic filesystem tasks and injection-driven context passing [7]. When onboarding or debugging workflow author intent, examples are often faster to reason through than raw schema docs because they expose composition tradeoffs explicitly [2][18]. This node is also the easiest place to seed new acceptance scenarios before formalizing them into AT IDs and tests [21][25].

[20] **Repository Topology and Packaging** - The repository has a compact top-level shape: `orchestrator/` runtime, `specs/` contracts, `tests/` verification, `workflows/examples/` authored samples, and `scripts/` helper tooling [1]. Packaging is minimal through `pyproject.toml` (`project.name=orchestrator`, Python >=3.11, dependencies on `pyyaml` and `jsonschema`) with optional dev extras for pytest tooling [18]. CLI entry code is in `orchestrator/cli/main.py` and module bootstrap in `orchestrator/__main__.py`, while package version is pinned in `orchestrator/__init__.py` [4]. The largest code concentration is `orchestrator/workflow/executor.py`, so repository complexity is functionally centralized even though modules are split by concern [5]. This topology supports a spec-first workflow: docs define behavior, runtime enforces behavior, tests verify behavior, and examples teach behavior [2][19].

[21] **Acceptance-ID Traceability Graph** - `specs/acceptance/index.md` is the canonical matrix of AT requirements and spans DSL validation, substitution, dependencies/injection, IO capture, providers, wait-for, state integrity, CLI safety, and security [2]. Code comments in runtime modules and test names mirror these IDs (for example AT-28..35 in injection paths and AT-56..59 in control flow), creating an explicit requirements-to-code map that is rare in small orchestration projects [5][10][11]. This traceability lowers debugging search cost: a failing behavior can often be routed to a specific AT cluster and then to the owning module/test pair quickly [12][18]. It also disciplines feature additions, because new behavior generally requires updating spec text, acceptance index, runtime logic, and tests together to preserve coherence [6][25]. In practice, node [21] is the fastest “index-to-implementation” bridge for contributors unfamiliar with the codebase [1].

[22] **Retries, Timeouts, and Failure Recovery Semantics** - Retry policy primitives are in `orchestrator/exec/retry.py` and are applied differently for command and provider steps: providers retry on exit codes 1/124 by policy, while commands retry only when `retries` is configured [9]. `WorkflowExecutor` then composes these outcomes with goto handlers and strict flow, so exhausted retries can still transfer control via `on.failure.goto` before global stop/continue behavior is applied [11][21]. Timeout normalization to exit 124 is shared across command/provider/wait paths, which keeps failure handling predictable during mixed workflows [3][15]. Resume and force-restart flags further shape recovery posture by deciding whether to continue from persisted state or start a new lineage [14]. This node is where reliability semantics converge: transport-level failures, business-logic branching, and operator-selected recovery mode all meet here [4][18].

[23] **Path Safety Enforcement Model** - Path safety is defense-in-depth across compile-time and runtime layers: loader validation rejects obvious absolute/parent-escape declarations, and runtime modules (`wait`, dependency resolver) re-check resolved paths and symlink boundaries [6][10][15]. CLI safety adds operational checks around cleanup/archive targets to prevent accidental destructive behavior outside workspace context [4]. Security spec language in `specs/security.md` clarifies scope: orchestrator-resolved paths are guarded, but child processes can still access what OS permissions allow, so system isolation remains a deployment concern [16]. This means bug triage for path incidents should identify whether failure occurred in DSL validation, runtime resolution, or subprocess behavior to choose the right fix layer [17][22]. Contributors adding new file-affecting features should route them through existing safety helpers rather than ad-hoc path logic [25].
